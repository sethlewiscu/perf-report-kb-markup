What?

This SOP outlines gathering data about, troubleshooting, creating, performance reports.

What IS a performance issue?

Examples of performance issues:

  • Slow loading times
  • Operations that lead to the platform or feature freezing
  • Slow initial loads (new tabs, opening the desktop app, refreshing the page, logging in, etc.)
  • Slow page (route) changes (i.e., moving within the platform once loaded)

What is NOT 🙅🏾‍♂️ a performance issue?

Sometimes clients refer to these items as "performance" issues. Issues with the following features are bugs and should follow the standard bug reporting/escalating processes.

  • If you think the customer is demonstrating acceptable loading times, reach out to @performance-smes or the @performance-tim
  • Long wait time for Automations to fire
  • Long wait times for actions to occur from third-party integrations (Zapier, Make, Tray.io , etc.)

  • Data sync/refresh issues such as

    • Status not updating in another user's browser
    • Needing to refresh to see changes

(Feel free to raise to @performance-tim if user very frustrated/highly impacted)

When?

  1. Whenever a user writes in with complaints of slow loading or poor speeds
  2. After being pinged in a Social escalation

  3. Whenever a sales or red account escalation requests help with a client experiencing a performance issue

Reminders

  1. After confirming a performance issue, refrain from attaching a report to a defect in the standard way via the Defects Master List .
  2. If your user reports performance issues for a specific browser, desktop app, or mobile you should still follow the process below and create a performance report inside the Performance Report List .

How?

🏅 Golden Rule

Screen recordings of the perceived performance issue(s) are most important.

If you have answers to the questions presented in the Troubleshooting::Slow Speeds::Performance questions macro, it's time to create a report.

Here are some do's and don'ts when preparing a performance report.

Do

  • Ensure the issue(s) reported is not a functional issue/bug. (See "What is NOT a performance issue?" above.)
  • If you see the user encountering egregiously slow loading, do sympathize with them . If they're frustrated AND if you see clear signs of latency, you can acknowledge what they're experiencing is not up to our standard.

Don't

  • Indicate that their poor performance is our fault. Many performance issues are a result of poor hardware, internet connections, IT systems, etc. Do not admit/indicate for certain that the problem is with ClickUp.
  • Ask or suggest broad t-shooting steps e.g. check network connection, clear cache, try incognito etc.
  • Attempt to repro in user's Workspace. You won't be on the same network, machine, shard, etc. with them so a 1-1 repro is not possible.

FAQs

  • What do I do if I don't have all the required details for the Form submission?

    • If you need to submit an incomplete Form due to an escalation or other extenuating circumstance, feel free to use placeholder text in the required fields. Just be sure to supply the details later as Eng relies on the information we provide!

  • What do we do when users report multiple performance issues but didn't share Clips of all the issues?

    • If the Clips the user did provide show clear performance issues (regardless of the feature area), create the report

    • Otherwise, continue t-shooting with the user until

      1. They share clear evidence of a performance issue; or
      2. You bug them up for a functional issue/bug
